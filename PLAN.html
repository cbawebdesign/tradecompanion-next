BUG 2. Catalyst PRs Not Coming in Real-Time Next

Problem: Jon reported MOBX catalyst PR at 8:00 didn't arrive until reboot. PRs are currently fetched only on load via DB backfill — the real-time SignalR subscription for PRs was never finished.

Root Cause: In useSignalR.ts, the hook subscribes to tradingViewAlertRaw, ScannerAlert, and BroadcastAlertTrigger events. But BroadcastAlertTrigger is only fired from the legacy desktop app's AlertService.cs — it's NOT fired from the Azure Functions backend where catalysts/PRs originate. The Azure Function uses a different SignalR hub.
Fix Options:
  1. Option A — Subscribe to Azure SignalR events directly. The Azure Function hub fires newsAlertRaw for PRs and catalystAlertRaw for catalysts. Add these handlers in useSignalR.ts on the Azure socket connection.
  2. Option B — Relay through the desktop app. The desktop AlertService already calls BroadcastAlertTrigger when PRs arrive. If TC desktop is running, PRs flow. But this makes next-gen dependent on legacy running.
  3. Option C — Poll the DB on an interval. Set a 30s timer to re-fetch /AlertsBySymbol. Simple but introduces latency.

Recommended: Option A — true real-time, no legacy dependency.

Files to change:
hooks/useSignalR.ts — add newsAlertRaw and catalystAlertRaw handlers
Need to confirm: exact event names on the Azure SignalR hub (check Azure Function code)

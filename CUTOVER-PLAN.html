◆ Web (Next.js)

── Sources (independent hooks) ──────────
useSignalR           → Quotes, TradingView,
                        TradeExch, Filings (newFiling),
                        BroadcastAlertTrigger

useFilingsPolling    → REST 30-60s
useTweetsPolling     → REST 30s

      │
      ▼ store.addAlert()
Zustand Store
      │
      ├─ Dedup: symbol + type + first 40 chars
      │
      ├─ alerts[] (max 500)
      │
      ├─ Audio: 800Hz beep (same for all types)
      │   No TTS. No per-type sounds.
      │
      ├─ BroadcastChannel → pop-out windows
      │
      ▼
AlertBar component → renders + scrolls


── Missing from pipeline ────────────────
× No FilteredPR keyword matching
× No RSS/MAIL feed polling
× No TradeExchangeFiltered (unfiltered)
× No TTS / per-type audio
× No Cosmos watchlist sync
Architecture difference: Legacy uses a centralized AlertService as a single router. Web uses decentralized hooks that each call store.addAlert() independently. The web approach is simpler and easier to extend, but lacks the central dedup/audio control that AlertService provides.

SEC 4 Cutover Action Plan — 5 Phases
⚠ Phase 1 — Alert Reliability Verification (CRITICAL)

Before adding anything new, verify that the alerts already wired up in the web version actually work reliably. Run legacy and web side-by-side during market hours and compare.

1.1 — Side-by-Side Test Session
  • Run legacy desktop + web version simultaneously on the same watchlist
  • For each alert that fires on legacy, confirm it also appears in web within acceptable latency
  • Log discrepancies: missed alerts, extra alerts, wrong symbols, wrong timestamps
  • Duration: Full trading day (pre-market through close)

VerifyAlert shows username, tweet text, link to tweet. Symbol correct

Alert Tagging System Planned

Justin wants alerts tagged on the backend so they arrive structured and filterable by symbol, type, source, and metadata. Currently alerts are loosely structured — each source builds its own alert shape. The tagging system standardizes this.

Current State (Unstructured)
Problem: Each alert source builds its own shape independently.
SignalR BroadcastNews arrives as: { symbol, headline, story_id, savetime_et }
SignalR newFiling arrives as:     { symbol, form, url, title, dcn, cik }
SignalR tradingViewAlertRaw:      { id, raw_text, received_utc }
REST    /api/tweets:              { username, text, id_long, created_at }

Each hook maps these differently into Alert { symbol, message, type, color, url }
Dedup happens at the store level on symbol + type + first 40 chars = fragile
No standardized metadata = can't filter by source, provider, or confidence
Target State (Tagged)
Solution: Standardized alert envelope with rich tags.

TaggedAlert {
  id:           string          // unique UUID
  symbol:       string          // primary symbol (uppercase)
  symbols:      string[]        // ALL mentioned symbols (multi-symbol tweets)
  type:         AlertType       // news | filing | tweet | catalyst | trade_exchange | tradingview | price
  source:       AlertSource     // signalr_azure | signalr_news | rest_poll | client_computed | auditor_recovery
  provider:     string          // TIGER | AMPS | SEC | X | TradingView | TradeExchange
  message:      string          // display text
  rawPayload:   any             // original untouched data from the source
  url:          string | null   // clickable link
  timestamp:    Date            // event time (ET)
  receivedAt:   Date            // when web client received it
  tags: {
    watchlist:  string | null   // which watchlist the symbol is on
    isFiltered: boolean         // matched FilteredPR keyword?
    filterType: 'positive' | 'negative' | null
    formType:   string | null   // SEC form type (4, SC 13G, etc.)
    priority:   'high' | 'normal' | 'low'  // FilteredPR match = high, etc.
    isRecovered: boolean        // came from AlertAuditor?
    dedupKey:   string          // explicit dedup key (replaces fragile 40-char match)
  }
  color:        string          // computed from type + tags
  audio:        { frequency: number, duration: number } | null
}
What This Enables
  • Filter by source: “Show me only SignalR real-time alerts” vs “only polling backfill”
  • Filter by watchlist: “Only alerts for symbols on my 'Day Trade' watchlist”
  • Priority sorting: FilteredPR keyword matches float to top, routine filings sink down
  • Better dedup: Explicit dedupKey per source instead of fragile message substring matching
  • Audit trail: rawPayload preserved — debug exactly what the backend sent
  • Latency tracking: timestamp vs receivedAt shows delivery lag per source
  • Recovery visibility: isRecovered flag shows which alerts the Auditor caught
  • Backend integration: Tags can be applied server-side (Azure Function tags before broadcast) or client-side (hooks tag on receipt)
Implementation Approach
  • Phase A (Client-side): Extend the Alert type to TaggedAlert. Each hook tags on receipt. Store dedup uses dedupKey. No backend changes needed
  • Phase B (Backend-side): Azure Function adds tags before SignalR broadcast. Standardized alert envelope across all events. Clients receive pre-tagged alerts
  • Phase C (UI): AlertBar gains filter/sort controls. Filter by type, source, watchlist, priority. Saved filter presets per user

New Systems Built

Alert Auditor (Self-Healing)

hooks/useAlertAuditor.ts

  • Polls /api/AlertsBySymbol for each watchlisted symbol every 60 seconds
  • Compares backend data against what the web app displayed
  • If backend has an alert the web missed → fires it immediately as a recovered alert
  • Only runs during market hours (Mon-Fri 4am-8pm ET)
  • Catches the exact bugs Justin reported (HCTI PR missing, etc.)
  • Logs: AlertAuditor: RECOVERED missed alert — HCTI news: ...

FilteredPR Keyword Matching

lib/filteredPr.ts

  • Full port of FilteredPR.cs regex logic
  • Syntax: comma = OR, & = AND, ! = NOT, * = wildcard
  • Runs on every BroadcastNews event in useSignalR
  • Fires additional color-coded alert: green for positive match, red for negative
  • Config: filteredPrPositive and filteredPrNegative in Settings

Per-Type Alert Sounds + TTS

lib/alertAudio.ts

  • Replaces single 800Hz beep with configurable frequency + duration per alert type
  • Default sounds: PRs 800Hz, Filings 600Hz, Tweets 900Hz, Catalyst 700Hz, TradingView 1000Hz, Price 1200Hz
  • Text-to-Speech via Web Speech API (female voice, matches legacy SpeechService.cs)
  • 30-second dedup (same text not spoken twice within 30s — same as legacy)
  • Config: ttsEnabled toggle + per-type alertSounds object

AHK Companion Server

companion/ahk-server.js + lib/ahk.ts

  • ~80 line Node.js server that runs on Justin's Windows machine
  • Web calls http://localhost:9876/run?symbol=AAPL
  • Companion launches AHK script via child_process.exec()
  • Matches legacy ScriptsController.cs → Process.Start() behavior
  • Falls back to clipboard copy if companion not running
  • Config: ahkEnabled, ahkUrl in Settings
  • Also has /health endpoint for status checks

Cosmos DB Watchlist Sync

hooks/useCosmosSync.ts

  • Saves watchlists + config to Cosmos DB via PUT /api/user/{userKey}
  • 3-second debounce after last change (won't hammer the API)
  • On load: if localStorage has default empty watchlist, restores from cloud
  • Syncs: watchlists, flagged symbols, alert subscriptions, theme, filter configs
  • Browser clear / new device = data recovered from cloud

ExcludeFilings Config

useSignalR.ts + useFilingsPolling.ts

  • Pipe-separated form types to skip (e.g., SC 13G|4|D)
  • Applied in BOTH SignalR real-time handler AND REST polling hook
  • Matches legacy Filings.cs ExcludeFilings behavior
  • Config: excludeFilings in Settings

Multi-Symbol Tweet Fix

hooks/useTweetsPolling.ts

  • Legacy X.cs has break after first symbol match = only 1 alert per tweet
  • Web version fixed: fires separate alert for EACH watchlisted symbol in a tweet
  • Tweet mentions $AAPL, $TSLA, $NVDA and all 3 are on watchlist → 3 alerts
  • Non-watchlisted symbols still show with first extracted symbol

New Config Fields

types/index.ts + store/useStore.ts

  • ttsEnabled — Text-to-speech toggle
  • excludeFilings — Pipe-separated form type exclusions
  • filteredPrPositive / filteredPrNegative — PR keyword filters
  • showAllTradeExchange — Show unfiltered TX posts
  • ahkEnabled / ahkUrl — AHK companion toggle + URL
  • alertSounds — Per-type frequency + duration + enabled

The Cascade (Before Fix)

Every 250ms: quotes update
  → persist serializes EVERYTHING to localStorage (including quotes!)
    → triggers cross-window storage events
      → usePriceAlerts effect fires (iterates ALL symbols)
        → may call addAlert → triggers AlertBar + Watchlist re-render
          → persist fires again ← LOOP

After fix: quotes update → Zustand store updates → only QuoteProcessor re-renders
  Everything else is isolated. No localStorage writes. No cascade.

Visual QA Agent (Claude Computer Use)

  • What: Claude opens TC web in a real browser during market hours
  • Watches: Alert bar for new alerts, quote prices updating, UI responsiveness
  • Tests: Click symbol → verify ribbon loads. Click alert → verify URL opens. Scroll → smooth?
  • Detects: Duplicate alert rows, frozen UI, flickering components, broken layouts
  • Screenshots: Captures anomalies with timestamps for review
  • Frequency: Runs autonomously during market hours (9:30am–4pm ET), checks every 5 minutes
  • Output: Timestamped log of every interaction + screenshots of any issues found

Alert Integrity Agent

  • What: Queries the Azure Function APIs directly to get the ground truth of what alerts SHOULD exist
  • Queries: /api/AlertsBySymbol for each watchlisted symbol every 60s
  • Compares: Backend data vs what the web app's alert log recorded
  • Reports:
    • MISSED — Backend has it, web didn't fire it
    • DELAYED — Web fired it >30s after backend received it
    • DUPLICATE — Web fired same alert more than once
    • MATCHED — Confirmed correct delivery
  • Note: Works alongside useAlertAuditor which already recovers missed alerts in real-time. This agent validates that the auditor is doing its job.

Performance Monitor Agent

  • What: Monitors browser performance metrics during a trading session
  • Tracks: Memory usage trend (heap size over time), frame drops, long tasks (>50ms), DOM node count
  • Detects: Memory leaks (heap growing unbounded), jank (frames >16ms), excessive DOM mutations
  • Baselines: Establishes normal operating metrics (e.g., “steady state is 120MB heap, 800 DOM nodes”)
  • Alerts: If memory exceeds 2× baseline or frame budget exceeded for >5s
  • Uses: Chrome DevTools Protocol via headless browser, or performance.measureUserAgentSpecificMemory()

Report Aggregator

  • What: Combines output from all 3 agents into a single daily report
  • Runs: Automatically at market close (4:30pm ET)
  • Contains:
    • Alert delivery score (matched / total)
    • List of missed alerts with root cause analysis
    • Visual bugs found (with screenshots)
    • Performance trend chart (memory, frame rate)
    • Comparison to previous day's report
  • Action: For deterministic bugs (e.g., dedup logic failing for a specific alert type), auto-generates a code fix and opens a PR
  • Delivery: HTML report + optional Slack/email notification

Key insight: The context layer is what makes these agents intelligent rather than just scripts. A script checks “did alert appear.” An agent with context checks “did the right alert appear, from the right source, with the right fields, within the right latency, for symbols on the right watchlist, without duplicating a previous alert from a different source.”
